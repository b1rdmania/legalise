# Audit Reconstruction

The audit log is not just a table of events. The product requirement is:

> Show me everything AI did on this matter.

The reconstruction view must be regulator-legible, PI-insurance-legible, and useful to a supervising lawyer.

## Dimensions

The audit API and UI must filter by:

1. module
2. user
3. model
4. document
5. gate
6. failed attempt
7. output artifact
8. external provider
9. date/time

These dimensions are V1, not later polish.

## Required Audit Fields

Extend `audit_entries` with:

- `module_id`
- `capability_id`
- `permission_grant_id`
- `gate_state`
- `advice_tier`
- `output_artifact_id`
- `external_provider`
- `cost_estimate_pence`
- `input_artifact_ids`
- `document_ids`
- `failure_code`
- `failure_message`

Existing prompt/response hashes stay.

## Click-Through Reconstruction

Opening an audit entry should show:

- action
- actor
- matter
- timestamp
- module and capability
- permission snapshot at time of action
- gate state at time of action
- advice tier
- model/provider
- token counts and cost estimate
- input documents/resources
- output artifact
- prompt hash
- response hash
- failure details if any
- related entries in the same invocation

## Invocation Grouping

Every module invocation should have an invocation id.

Example:

```text
invocation_id = inv_...
```

All rows emitted during the same run share it:

- `module.invoked`
- `mcp.resource.read`
- `model.call`
- `document.generated`
- `module.completed`

This makes reconstruction coherent.

## Failure Semantics

Failures are audit events.

Rows must persist even if the request transaction rolls back. Existing `audit_failure()` is the pattern.

Minimum failure events:

- provider key missing
- provider upstream error
- storage write/read failure
- capability denied
- gate blocked
- sandbox denied
- dependency missing
- module timeout
- module parse failure

## Cost Tracking

Every model invocation records:

- provider
- model
- input tokens
- output tokens
- cost estimate
- currency/source of pricing table

Cost estimates are approximate and should be marked as such in UI.

Required summaries:

- per module
- per matter
- per user
- per provider
- per date range

## Storage Strategy

V1:

- single Postgres `audit_entries` table
- append-only by application convention and deploy-time database permissions
- no separate event-log service

Later:

- hash chain or external append-only event log
- regulator export package
- WORM storage class where deployed

## API Shape

```text
GET /api/matters/{slug}/audit
GET /api/matters/{slug}/audit/{entry_id}
GET /api/matters/{slug}/audit/summary
```

Filter query:

```text
?module=&user=&model=&document=&gate=&failed=&output=&provider=&from=&to=&invocation_id=
```

## UI Requirements

The audit reconstruction UI must support:

- dense table
- filter sidebar
- detail drawer
- invocation grouping
- failed-attempt toggle
- model/cost summary
- export current view

It must not hide failed or blocked attempts by default.

## Completeness Checks

Phase 16 must include scenarios:

- all AI events on Khan matter
- all failed attempts
- all events touching one document
- all model calls by provider
- all outputs generated by one module
- all gate-blocked actions
