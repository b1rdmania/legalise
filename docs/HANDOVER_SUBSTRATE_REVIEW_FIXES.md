## 2. Review Findings

### P1. Matter delete can succeed while storage cleanup fails

**File:** `backend/app/api/matters.py`

Current shape:

- `DELETE /api/matters/{slug}` writes `matter.deleted` and maybe `matter.deleted_without_export`.
- It calls `storage.delete_prefix(matter_prefix(user.id, matter.id))`.
- It catches `Exception`, swallows it, sets `matter.status = archived`, commits, and returns `204`.

Why this is a blocker:

- The endpoint is part of the deletion / retention hardening work.
- Returning success while R2/MinIO deletion failed leaves uploaded/generated bytes behind.
- That is worse than not having deletion, because the user/operator receives a false success signal.

Implement one of these shapes:

**Recommended for now: fail closed.**

- If `storage.delete_prefix()` raises, do **not** archive the matter.
- Roll back the transaction.
- Return `502` or `503` with structured detail:
  - `error: "matter_storage_delete_failed"`
  - `message`
  - optionally `matter_slug`
- Write an audit row only if it is safe and transactionally coherent. Do not write `matter.deleted` if the delete did not happen.

Alternative if the builder strongly prefers retry semantics:

- Add a matter status such as `delete_failed` or `pending_delete`.
- Store the failure in an audit row.
- Provide a retry path.

Do not silently pass.

Acceptance:

- Test storage backend raising from `delete_prefix`.
- API must not return `204`.
- Matter remains visible/not archived after the failure.
- No `matter.deleted` success row is written on failed storage cleanup.
- Happy path still archives the matter and deletes storage prefix.

### P1. Job creation can strand queued jobs if Redis enqueue fails

**Files:**

- `backend/app/api/jobs.py`
- `backend/app/api/exports.py`
- `backend/app/core/jobs.py`
- `backend/app/worker.py`

Current shape:

- Job row is created and committed.
- `_enqueue_job()` then pushes the job id to arq/Redis.
- In `jobs.py`, Redis enqueue failure raises after the queued row is already durable.
- In `exports.py`, Redis enqueue failure is swallowed with a comment saying the worker will pick it up later.
- The worker only runs jobs that arq receives. There is no DB poller or outbox sweeper.

Why this is a blocker:

- A failed enqueue leaves a permanent `queued` job.
- That job consumes the active-job limit.
- The user sees either a failed request with a hidden durable row or a successful export request that never runs.

Implement one of these shapes:

**Recommended for now: enqueue failure marks the job failed and returns a clear error.**

- Keep the current job-row-first shape if desired.
- If `_enqueue_job()` fails:
  - reopen/update the job row in the same request session where possible.
  - set `status = failed`.
  - set `error_code = "enqueue_failed"`.
  - set `error_message` to a scrubbed message.
  - commit that terminal state.
  - return `503` with `error: "job_enqueue_failed"`.
- Ensure active-job count no longer includes the failed job.

Alternative:

- Add a real outbox/re-enqueue sweeper.
- If you do this, document and test the sweeper. Do not just comment that it exists.

Acceptance:

- Test `_enqueue_job()` raising for Pre-Motion job creation.
- Test `_enqueue_job()` raising for Contract Review job creation.
- Test `_enqueue_job()` raising for Export job creation.
- Each response is a structured 503 or equivalent, not a silent success.
- Persisted job state is terminal `failed` with `error_code = "enqueue_failed"`.
- Active-job limit is not consumed by the failed enqueue.

### P1. Tabular Review `.docx` export is still filesystem-backed

**Files:**

- `backend/app/modules/tabular_review/export.py`
- `backend/app/api/documents.py`

Current shape:

- `tabular_review/export.py` writes the `.docx` to `settings.matters_root`.
- It records `storage_uri = generated/...`.
- The generated document download endpoint now reads generated files via `get_storage_backend()`.

Why this is a blocker:

- In production/S3, the tabular review export is recorded but not actually stored in object storage.
- The download endpoint will look in R2/MinIO and fail.
- This contradicts the Unit 1 storage acceptance bar that generated artefacts no longer rely on Fly local disk as source of truth.

Fix:

- Change `export_review_docx()` to write bytes via `get_storage_backend().put_bytes()`.
- Use `generated_key(user_id=matter.created_by_id, matter_id=matter.id, document_id=file_uuid, filename=...)` or an equivalent canonical key.
- Keep the existing `document.generated` and `module.tabular_review.exported` audit rows, but make their `storage_uri` point to the object-storage key.

Acceptance:

- Test `export_review_docx()` writes to `LocalStorageBackend` in tests.
- Test `/api/documents/generated/{file_uuid}` can download the tabular review export.
- Test no write to `settings.matters_root` is required for the export path.

### P2. Workflow-run daily limit is declared but not enforced or exposed

**Files:**

- `backend/app/core/limits.py`
- `backend/app/core/jobs.py`
- `backend/app/api/jobs.py`
- `backend/app/api/exports.py`
- `backend/app/api/usage.py`
- `backend/tests/test_limits.py`
- `backend/tests/test_jobs.py`

Current shape:

- `workflow_runs_per_day` exists in `Limits`.
- The handover says 50 runs/day is enforced.
- Actual enforcement is only active-job count.
- `/api/me/usage` omits `workflow_runs_today` and `active_jobs`.

Fix:

- Add a `check_workflow_run(user_id, session)` helper.
- Define exactly which job kinds count as workflow runs:
  - likely `pre_motion` and `contract_review`.
  - decide whether `export` counts. Reviewer view: export should probably **not** count as workflow run; it is data portability.
- Call the helper before creating counted jobs.
- Add `workflow_runs_today` and `active_jobs` to `UsageResponse`.

Acceptance:

- Test reaching the daily workflow-run cap returns `429 evaluation_limit_reached`.
- Test export jobs do or do not count according to the chosen policy.
- Test `/api/me/usage` exposes the workflow-run and active-job fields.

### P2. Matter export is not complete enough for the current claim

**File:** `backend/app/core/exports.py`

Current export includes:

- `matter_metadata.json`
- document metadata
- uploaded document bytes when `Document.storage_uri` exists
- `audit.json`
- `jobs.json`

Missing from a real "user can leave" export:

- chronology events
- document bodies (`DocumentBody`)
- document versions (`DocumentVersion`)
- document edits
- generated artefact bytes
- matter citations
- tabular reviews and rows
- possibly assistant messages, depending on product policy

There are two acceptable paths:

**Path A - Narrow the claim for v0.4.**

- Rename copy/comments/docs to "basic matter export bundle".
- Make clear it includes matter metadata, upload metadata/bytes, audit, jobs.
- Do not imply it is complete data portability yet.

**Path B - Complete the export now.**

- Add the missing rows/files above.
- Include generated artefact bytes by following `document.generated` audit rows and their `storage_uri`.
- Add tests that inspect zip contents.

Reviewer recommendation:

- If launch pressure matters, do Path A now and ticket Path B for v0.5.
- If the builder is already in export code, add chronology + document bodies + document versions now, because those are cheap and important.

Acceptance for Path A:

- Comments and handover/docs no longer call it complete export/delete.
- Tests assert the exact included files so the scope is explicit.

Acceptance for Path B:

- Tests assert the extra content is included in the zip.
- Missing storage object should be recorded in a manifest warning, not silently disappear.
